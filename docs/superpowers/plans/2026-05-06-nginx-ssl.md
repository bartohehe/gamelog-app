# Nginx Reverse Proxy + Automatic SSL — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add an Nginx reverse proxy to the Docker Compose stack that routes all traffic through a single entry point, with optional automatic Let's Encrypt SSL configured via a `DOMAIN` env var.

**Architecture:** A custom `nginx` service (nginx:alpine + certbot in one image) sits in front of both `frontend` and `backend`. An `entrypoint.sh` script decides at startup whether to run HTTP-only (DOMAIN empty) or request/renew a Let's Encrypt cert and serve HTTPS (DOMAIN set). Frontend and backend lose their host ports; only nginx is exposed.

**Tech Stack:** Docker Compose, nginx:alpine, certbot (certbot-nginx/webroot), envsubst, shell scripting

## Task 1: Fix VITE_API_URL so frontend uses relative paths through nginx

> **Why first:** `api.ts` has `baseURL: import.meta.env.VITE_API_URL ?? 'http://localhost:5000'`. When nginx proxies `/api/*` to the backend, the frontend must call `/api/…` (relative path). An empty `VITE_API_URL` produces an empty string, which `??` passes through as `''` — axios then uses relative paths. Without this fix, the browser-side JS calls `localhost:5000` or `localhost:8081`, which don't exist on the user's machine.

**Files:**
- Modify: `docker-compose.yml` (frontend build args)
- Modify: `.env`

- [ ] **Step 1: Open `docker-compose.yml` and find the frontend build args**

Current state in `docker-compose.yml`:
```yaml
frontend:
  build:
    context: ./frontend
    args:
      VITE_API_URL: http://localhost:8081
```

- [ ] **Step 2: Replace the hardcoded URL with an env-var passthrough**

```yaml
frontend:
  build:
    context: ./frontend
    args:
      VITE_API_URL: ${VITE_API_URL:-}
```

The `:-` syntax means: use the env var if set, default to empty string. Empty string → axios relative paths → all `/api/*` calls go through nginx on the same origin.

- [ ] **Step 3: Add VITE_API_URL to `.env`**

Append to `.env`:
```
# Frontend API URL — leave empty when using nginx reverse proxy
# Set to http://localhost:8081 for local dev without nginx
VITE_API_URL=
```

- [ ] **Step 4: Commit**

```bash

git commit -m "feat: make VITE_API_URL configurable for nginx reverse proxy"
```

## Task 3: Create nginx config templates

**Files:**
- Create: `docker/nginx/templates/http.conf.template`
- Create: `docker/nginx/templates/https.conf.template`

### http.conf.template

Used in two scenarios:
1. `DOMAIN` is empty — permanent HTTP-only mode
2. `DOMAIN` is set — temporary mode while certbot completes the ACME HTTP-01 challenge

The `/.well-known/acme-challenge/` location is always present (harmless when certbot isn't running).

- [ ] **Step 1: Create `docker/nginx/templates/http.conf.template`**

```nginx
server {
    listen 80;
    server_name _;

    # ACME HTTP-01 challenge — used by certbot during cert issuance
    # Harmless when DOMAIN is not set
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    location /api/ {
        proxy_pass         http://backend:8080;
        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
    }

    location / {
        proxy_pass         http://frontend:80;
        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
    }
}
```

> **Note on `envsubst`:** This template contains no `${VARIABLE}` placeholders — it uses nginx variables like `$host` directly. It is copied as-is. Only `https.conf.template` uses `${DOMAIN}` substitution.

### https.conf.template

Two server blocks:
- `:80` — serves ACME challenge, redirects everything else to HTTPS
- `:443` — TLS termination with Mozilla Modern cipher suite, proxies to backend/frontend

- [ ] **Step 2: Create `docker/nginx/templates/https.conf.template`**

```nginx
# HTTP → HTTPS redirect (also serves ACME challenge for cert renewal)
server {
    listen 80;
    server_name ${DOMAIN};

    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    location / {
        return 301 https://$host$request_uri;
    }
}

# HTTPS — TLS termination + reverse proxy
server {
    listen 443 ssl;
    server_name ${DOMAIN};

    ssl_certificate     /etc/letsencrypt/live/${DOMAIN}/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/${DOMAIN}/privkey.pem;

    # Mozilla Modern TLS configuration
    ssl_protocols             TLSv1.2 TLSv1.3;
    ssl_ciphers               ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:DHE-RSA-AES128-GCM-SHA256:DHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;
    ssl_session_timeout       1d;
    ssl_session_cache         shared:MozSSL:10m;
    ssl_session_tickets       off;

    location /api/ {
        proxy_pass         http://backend:8080;
        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
    }

    location / {
        proxy_pass         http://frontend:80;
        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
    }
}
```

## Task 4: Create entrypoint.sh

**Files:**
- Create: `docker/nginx/entrypoint.sh`

The script is PID 1 in the container. It decides mode based on `$DOMAIN`:

**HTTP-only mode (DOMAIN empty):**
Copy http template → `/etc/nginx/conf.d/default.conf` → `exec nginx` (replaces shell as PID 1).

**HTTPS mode (DOMAIN set):**
1. Generate HTTP config, start nginx in background
2. Run certbot if cert doesn't exist yet
3. Generate HTTPS config, reload nginx
4. Loop every 12h: renew cert → reload nginx

- [ ] **Step 1: Create `docker/nginx/entrypoint.sh`**

```sh
#!/bin/sh
set -e

# ── HTTP-only mode (no DOMAIN configured) ────────────────────────────────────
if [ -z "$DOMAIN" ]; then
    echo "[nginx] DOMAIN not set — running in HTTP-only mode"
    cp /etc/nginx/templates/http.conf.template /etc/nginx/conf.d/default.conf
    exec nginx -g 'daemon off;'
fi

# ── HTTPS mode ────────────────────────────────────────────────────────────────
if [ -z "$SSL_EMAIL" ]; then
    echo "[nginx] ERROR: SSL_EMAIL is required when DOMAIN is set"
    exit 1
fi

echo "[nginx] DOMAIN=$DOMAIN — starting with Let's Encrypt SSL"

# Phase 1: start nginx on HTTP so certbot can complete the ACME challenge
cp /etc/nginx/templates/http.conf.template /etc/nginx/conf.d/default.conf
nginx -g 'daemon off;' &
NGINX_PID=$!

# Phase 2: obtain certificate (skip if already present from a previous run)
if [ ! -f "/etc/letsencrypt/live/${DOMAIN}/fullchain.pem" ]; then
    echo "[certbot] Requesting certificate for $DOMAIN..."
    certbot certonly \
        --webroot \
        --webroot-path=/var/www/certbot \
        --email "$SSL_EMAIL" \
        --agree-tos \
        --no-eff-email \
        --non-interactive \
        -d "$DOMAIN"
    echo "[certbot] Certificate obtained successfully"
fi

# Phase 3: switch nginx to HTTPS config
envsubst '${DOMAIN}' < /etc/nginx/templates/https.conf.template > /etc/nginx/conf.d/default.conf
nginx -s reload
echo "[nginx] Now serving HTTPS for $DOMAIN"

# Phase 4: renewal loop — runs every 12 hours
# certbot renew is a no-op if cert has more than 30 days remaining
trap "kill $NGINX_PID; exit 0" TERM INT
while true; do
    sleep 12h &
    wait $!
    echo "[certbot] Running renewal check..."
    certbot renew \
        --quiet \
        --webroot \
        --webroot-path=/var/www/certbot
    nginx -s reload
    echo "[nginx] Config reloaded after renewal check"
done
```

> **Why `envsubst '${DOMAIN}'`:** Passing the variable list to `envsubst` prevents it from replacing nginx variables like `$host`, `$scheme`, `$request_uri` — only `${DOMAIN}` is substituted.

> **Why `sleep 12h & wait $!`:** This pattern allows the `trap` to interrupt the sleep when Docker sends `SIGTERM`, avoiding a 12-hour shutdown delay.

## Self-Hosted HTTPS Setup Reference (in README / docs)

For a self-hoster with a domain, the full flow is:

```bash
# 1. Point your domain's A record to the server IP (do this first!)
# 2. Open ports 80 and 443 in your firewall/router
# 3. Set env vars in .env:
DOMAIN=gamelog.example.com
SSL_EMAIL=admin@example.com

# 4. (Re)build and start — certbot runs automatically on first start
docker compose up -d --build

# 5. Verify
curl https://gamelog.example.com/api/featureflags
```

Cert renewal runs automatically every 12 hours inside the container. No cron job needed on the host.
